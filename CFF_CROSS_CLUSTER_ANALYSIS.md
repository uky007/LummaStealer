# CFF クロスクラスター構造の分析

**日付**: 2026-03-21
**分析者**: uky + Claude Opus 4.6

---

## 発見の概要

4つのCFFクラスター（Cluster 0-3）は独立した関数ではなく、**1つの巨大なCFF保護関数の部分**であることが判明した。CFF制御フローがクラスター間をまたいでジャンプし、レジスタ値を受け渡す。

これにより、Cluster 0/1 のテーブルエントリが有効な .text アドレスではない理由が解明された。

---

## 問題の経緯

### 当初の観察

CFFディスパッチャーのジャンプテーブルを .data セクションから読み取ったところ：

| クラスター | テーブルアドレス | エントリの内容 |
|-----------|----------------|---------------|
| Cluster 0 | 0x028530F0 | 0x2B882412, 0xBD8EA00A... (有効アドレスではない) |
| Cluster 1 | 0x028532D0 | 0x62AFA4D9, 0x5B3B48EC... (有効アドレスではない) |
| Cluster 2 | 0x02854600- | 0x0281E64A, 0x0281E678... (**有効な .text アドレス**) |
| Cluster 3 | 0x028547E0 | 0x0281D188, 0x0281D284... (**有効な .text アドレス**) |

Cluster 2/3 のテーブルは直接読めたが、Cluster 0/1 は「暗号化されている」と仮定した。

### 調査した仮説と結果

| 仮説 | 検証方法 | 結果 |
|------|---------|------|
| テーブルがXOR/ADD暗号化 | ブルートフォースキー探索 | 有効なキーなし |
| ローダーがテーブル復号 | WriteProcessMemoryダンプ比較 | テーブル完全一致（変更なし） |
| エントリポイントで復号 | Unicorn 100K命令エミュレート | テーブルへの書き込みゼロ |
| .relocで修正 | リロケーション解析 | .dataにリロケーションゼロ |
| デコード命令がある | mov-jmp間の命令走査 | eaxを変更する命令なし |
| 死んだコード | クロスリファレンス | **呼び出し元あり（否定）** |

全ての仮説が否定された。

---

## 解明：クロスクラスタージャンプ

### 決定的な証拠

IDA上で `0x02827467` (Cluster 0 内) に **外部からの xref** を発見：

```
.text:0284040B    jmp     loc_2827467     ; Cluster 3 → Cluster 0 へジャンプ
.text:02840410    nop                      ; (CFFパッチ後のNOP)
.text:02840411    nop
```

`0x0284040B` は **Cluster 3 の範囲** (0x0283A9CE-0x02840A69) 内にある。

### 制御フローの実際の動作

```
┌─────────────────────────────────────────────────┐
│ Cluster 3 のコード                                │
│   ... 何らかの処理で eax に有効アドレスを設定 ...    │
│   0x0284040B: jmp loc_2827467                     │
└──────────────────────┬──────────────────────────-─┘
                       │ eax = 有効な .text アドレス
                       ▼
┌─────────────────────────────────────────────────┐
│ Cluster 0 のテーブル初期化コード                     │
│   0x02827467: mov [esp+X], offset unk_28530F0    │
│   0x02827472: mov [esp+X], 0Ch                   │
│   ...（70+ 命令の mov [esp+X], imm）...            │
│   ※ eax は一切変更されない                         │
│   0x028276B9: jmp eax  ← Cluster 3 の値で正しくジャンプ │
└─────────────────────────────────────────────────┘
```

### なぜ線形解析で見えなかったか

線形コードフロー（関数エントリから順次実行）では：

```
0x028272B9: sub esp, 0x348          ; 関数プロローグ
...
0x028273F3: mov eax, [esp+0x2b0]    ; eax = 4 (index)
0x028273FA: mov ecx, [esp+0x2b4]    ; ecx = 0x028530F0 (table)
0x02827401: mov eax, [ecx+eax*4]    ; eax = table[4] = 0xBD8EA00A ★無効！
0x02827404: mov [esp+X], ...        ; テーブル初期化開始
...                                  ; 70+ 命令 (eax 変更なし)
0x028276B9: jmp eax                 ; jmp 0xBD8EA00A → クラッシュ？
```

しかし実際のCFF制御フローでは、`0x02827467` へはCluster 3から`jmp`で到達する。
この時 eax は Cluster 3 のディスパッチャーが設定した**有効なアドレス**を持っている。

### テーブル (0x028530F0) の真の役割

テーブルエントリ（0x2B882412, 0xBD8EA00A 等）は**ジャンプターゲットではない**。

テーブル初期化コードの構造：
```
mov [esp+0x2b4], 0x028530F0    ; テーブルベースポインタ
mov [esp+0x2b0], 4              ; パラメータ
...
mov [esp+0x2a4], 0x028530F0    ; 別のペア
mov [esp+0x2a0], 6              ; 別のパラメータ
...
```

これらのペア（ポインタ + 整数）は**CFFルーティングテーブルの設定データ**。
各ペアが1つのCFFディスパッチャーに対応し、「このディスパッチャーはテーブルXのインデックスYを使え」という指示。

`0x02827401` の `mov eax, [ecx+eax*4]` は、この初回セットアップの一部として
テーブルから値を読むが、その値がジャンプに直接使われるわけではない
（eax はクロスクラスタージャンプで上書きされる）。

## CFF全体構造

### 4クラスターの関係

```
                    ┌──────────────┐
                    │  Cluster 0   │
                    │ 0x028272B9-  │
              ┌────→│ 0x02828921   │←────┐
              │     └──────────────┘     │
              │                           │
  ┌──────────────┐                 ┌──────────────┐
  │  Cluster 1   │                 │  Cluster 3   │
  │ 0x0282A3C0-  │                 │ 0x0283A9CE-  │
  │ 0x0282DF72   │                 │ 0x02840A69   │
  └──────────────┘                 └──────────────┘
              │                           │
              │     ┌──────────────┐     │
              └────→│  Cluster 2   │←────┘
                    │ 0x02835C40-  │
                    │ 0x02836F5D   │
                    └──────────────┘
```

確認済みリンク：
- Cluster 3 (0x0284040B) → Cluster 0 (0x02827467)

推定：他のクラスター間にも同様のリンクが存在する可能性が高い。

### 「巨大CFF関数」の全体像

| 属性 | 値 |
|------|-----|
| 関数エントリ | 0x028272B9 (Cluster 0) |
| 全体範囲 | 0x028272B9 - 0x02840A69 |
| 全体サイズ | 0x197B0 = 104,368 バイト (約102KB) |
| ディスパッチャー数 | 467 (全クラスター合計) |
| テーブル種類 | 2種類（平文: Cluster 2/3, ルーティングデータ: Cluster 0/1） |

## 分析手法のまとめ

### 何がうまくいったか

1. **Capstone静的スキャン**: 467ディスパッチャーを全検出（contiguous + split）
2. **平文テーブル読み取り**: Cluster 2/3 のテーブルから244/322ディスパッチャー解決
3. **jcc/jmp書き換え**: if/else分岐構造の復元（itoa, wtoa等）

### 何が難しかったか

1. **Cluster 0/1テーブルの解釈**: 線形解析ではジャンプターゲットに見えるが実際は異なる
2. **クロスクラスター構造の発見**: IDAのxref解析で初めて判明
3. **「暗号化テーブル」仮説の検証と否定**: 5種類の検証で全て否定 → 正しい構造の発見に至る

### 今後の改善方針

Cluster 0/1 の残りディスパッチャーを解決するには：
1. クロスクラスタージャンプを全て特定（Cluster 3 → 0, 0 → 1, etc.）
2. 各ジャンプ時点の eax 値を Unicorn で追跡
3. CFF制御フローグラフ全体を再構築
